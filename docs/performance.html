linkPerformance

To obtain a proper analysis of RxDeep's performance (and generally complexity of the problem it tries to solve), we need to establish a few definitions first. Assume we have a state tree looking like this:

1{                                 // --> root of the state-tree

2  name: 'Cool Team',              // --> a leaf node

3  people: [                       // --> a middle node

4    {                             // --> a middle node

5      name: 'Jerome',             // --> a leaf node

6      age: 42,                    // --> a leaf node

7      address: {                  // --> a middle node

8        city: 'Jersey',           // --> a leaf node

9        country: 'Jibuti'         // --> a leaf node

10      },

11    },

12    {                             // --> a middle node

13      name: 'Jacqueline',         // --> a leaf node

14      age: 27,                    // --> a leaf node

15      address: {                  // --> a middle node

16        city: 'Jakarta',          // --> a leaf node

17        country: 'Jamaica'        // --> a leaf node

18      }

19    }

20  ]

21}

Two parameters of this tree are particularly important to analyzing complexity of the reactive change propagation problem:

  • Number of leaf nodes, in this case 9 (name of the team, name and age of each person, and the city and country of their address). We denote this number by letter nnn.

  • Depth of the tree, in this case 4 (you need 4 property resolutions to get from the root of the object to the deepest leaf node). We denote this number by letter ddd.

We will express complexity and performance of algorithms using the Big O notation, which is displayed with O(f(n,d))\Omicron(f(n, d))O(f(n,d)) syntax, since the complexity is always expressed as a function of nnn and ddd.

Parameters nnn and ddd are independent. State-tree can be an array with a thousand strings, where we would have nnn = 1000, ddd = 1, or it can be a single string nested in 1000 arrays ([[[...['x']...]]]), where we would have nnn = 1, ddd = 1000. We say a state-tree is typical, if ddd is in the order of log⁡(n)\log(n)log(n), i.e.

O(d)=O(log⁡(n)).\Omicron(d) = \Omicron(\log(n)).O(d)=O(log(n)).1

In practice, most of the time you have lower bounds and upper bounds on average number of childs each node of the state-tree has, which would mean that in practice most state-trees are typical.

linkChange in Leaf Nodes

For changes in a single leaf node, the node itself should emit a change, its parent should emit a change, the parent of its parent should emit a change, and all the path from the node to the root state should emit changes:

1const root = [{address: { city: 'Jakarta' }}];

2const mid1 = root.sub(0);

3const mid2 = mid1.sub('address');

4const leaf = mid2.sub('city');

5

6leaf.value = 'Jacksonville';

7

8// Now leaf, mid2, mid1 and root must emit changes.

This means that each change in a leaf node must result in at least O(d)\Omicron(d)O(d) operations, independent of the underlying algorithm. For a typical state-tree, that would be O(log⁡(n))\Omicron(\log(n))O(log(n)) operations.

For each such change, RxDeep performs one assignment by reference (to update local state's value) without emission, and constructs a trace object, passing the change with the trace object to the upstream. This would result in O(d)\Omicron(d)O(d) operations until the change reaches the root of the state-tree (alongside O(d)\Omicron(d)O(d) trace objects).

The root state then typically bounces the change back. Each recepient state will emit the new value, and perform equality checks between all of its generated sub-keys and head of received trace. On match, it will extract the downstream trace and pass down the resulting change to the matching sub-key. Assuming an average number of kkk sub-states per state, this yields O(kd)\Omicron(kd)O(kd) equality checks and O(d)\Omicron(d)O(d) emissions until the change is fully propagated across the state-tree.

Since kkk is a constant, this overall yields O(d)\Omicron(d)O(d) operations to fully propagate each change across the state-tree, O(log⁡(n))\Omicron(\log(n))O(log(n)) for typical state-trees, matching the lower complexity bound for any solution.

linkKeyed States

A Keyed State will track changes based on mapping of objects of it content array to particular keys using a given key function instead of using indexes. While doing so, it also produces detailed ListChange objects outlining exactly how the array has changed in terms of additions, deletions and items being moved from a particular index to another.

For doing so, KeyedState conducts O(n)\Omicron(n)O(n) operations, also maintaining a key map of O(n)\Omicron(n)O(n) size holding quick-access references by keys to elements producing those keys for the previous value. Here, nnn denotes not the number of leafs of the whole state-tree but the number of elements of the array assigned to a particular KeyedState.

For each change to the array, all elements are mapped once to their corresponding keys using the provided key function. The key function is supposed to be extremely light in nature, fetching a particular parameter of each element or doing a simple and fast computation:

1// a key function should be something in lines of these:

2p => p.id;

3n => n % 17;

If the key function does not have complexity of O(1)\Omicron(1)O(1), say O(λ)\Omicron(\lambda)O(λ) for some non-constant λ\lambdaλ, then upon each change the KeyedState would be conducting O(λn)\Omicron(\lambda n)O(λn) operations.

When indexed items in the array are moved around, the KeyedState also conducts partial retracing to generate a change trace invariant to its keys. This does not involve invoking the key function, but worst-case can add O(nlog⁡(n))\Omicron(n\log(n))O(nlog(n)) operations to change propagation process.

This worst case only occurs when all of the items of the array are shifted and their corresponding values are also mutated all the way to the leafs. In a typical case when the items of the array are moved without being modified themselves, then this lowers to O(n)\Omicron(n)O(n) operations.
