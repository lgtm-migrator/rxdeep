linkPrecision

RxDeep is designed to be extremely precise, meaning State objects should emit only when they have a good reason to. To be more precise (pun unintended), absolute precision means a State emits values when and only when one of the following holds:

  1. Its value has changed.

  2. Its value is directly updated (possibly to the same value).

  3. It points to the same address in state-tree as another State whose value is directly updated (possibly to the same value).

  4. One of its descendent sub-states (a sub-state in its sub-tree) satisfies (2) or (3).


1const root = new State([

2  { name: 'Jack', address: { city: 'Java', country: 'Indonesia' } },

3  { name: 'Jafar', address: { city: 'Jiroft', country: 'Iran' } },

4]);

5

6const mid1 = root.sub(0);

7const mid2A = mid1.sub('address');                     // --> shares tree address with mid2B

8const mid2B = root.sub(0).sub('address');              // --> shares tree address with mid2A

9const leaf1 = root.sub(0).sub('address').sub('city');  // --> shares tree address with leaf2

10const leaf2 = mid2A.sub('city');                       // --> shares tree address with leaf1

11

12leaf1.value = leaf1.value;                             // --> issue update without value change

13

14// As a result:

15// > leaf1 will emit (due to (2))

16// > leaf2 will emit (due to (3))

17// > mid2A, mid2B, mid1, and root will emit (due to (4))


By this definition RxDeep is absolutely precise if object immutabilty is respected and objects passed to it are of plain (non-circular) JavaScript objects.

linkLossiness

A Loss is when a State should emit (according to the outlined criteria) but it doesn't. RxDeep is not lossy by default, only IF object immutability is respected. Simply put, you must not change the value of a State without changing its reference, as otherwise there is no mechanism for RxDeep to pickup those changes and distinguish what has changed.

linkRedundancy

Redundancy refers to situations when a State emits without a good reason for doing so (i.e. none of the outline criteria hold up). RxDeep has no redundancy, meaning a State does not emit values without a good reason (without one of the aforementioned criteria being true).

This is due to the fact that leaf-changes are fully traced and delivered only to affected sub-states, and arbitrary changes are traced until their issuing depth and then post-traced so that the change trace is complete until leafs of the state-tree, allowing for precise propagation of the change.

linkPerformance

As discussed here, a leaf-change has O(log⁡(n))\Omicron(\log(n))O(log(n)) complexity and an arbitrary change at depth δ\deltaδ and above a sub-tree of nδn_{\delta}nδ​ nodes has O(log⁡(n)+nδlog⁡(nδ))\Omicron(\log(n) + n_{\delta}\log(n_{\delta}))O(log(n)+nδ​log(nδ​)) time complexity. For most day to day use cases both of these are more than fast enough.

However, in special cases you might need that additional performance. As (also) discussed here, if the whole sub-tree needs to change, you need minimum of O(log⁡(n)+nδlog⁡(nδ))\Omicron(\log(n) + n_{\delta}\log(n_{\delta}))O(log(n)+nδ​log(nδ​)) operations, as you need that many emissions to not be lossy. However, if your change affects a bounded number of leaf states, for example kkk leaf states, then the minimum number of operations is given by:

O(log⁡(n)+(k−1)log⁡(nδ))=O(log⁡(n))\Omicron(\log(n) + (k - 1)\log(n_{\delta})) = \Omicron(\log(n))O(log(n)+(k−1)log(nδ​))=O(log(n))1

A simple method of achieving that performance is identifying the kkk leaf nodes and applying change to them. Time-complexity of this solution is given by:

O(klog⁡(n))=O(log⁡(n))\Omicron(k\log(n)) = \Omicron(\log(n))O(klog(n))=O(log(n))1

However, this also results in kkk emissions by all affected states (e.g. the root state will also emit kkk times).

A more efficient solution would be to:

  1. Identify the top-most common ancestor of all affected leaf nodes,

  2. Apply changes respecting maximal object immutability.

1const company = new State({

2  teams: [{

3    people: [{name: 'Jack', age: 42}, {name: 'Jill', age: 31}],

4    name: 'Awesome Team',

5  }, ...]

6});

7

8//

9// This is the top-most common ancestor:

10//

11const target = company.sub('teams').sub(0);

12

13//

14// Now lets apply changes with maximal object immutability:

15//

16target.value = {

17  ...target.value,

18  name: 'Pro Team',

19  people: [

20    {

21      ...target.value.people[0],

22      name: 'Jafar'

23    }

24    ...target.value.people.slice(1),

25  ]

26}

Object immutability means that the reference of an object is changed IF its value has changed.
Maximal object immutability means the reference of an object is changed IF AND ONLY IF its value has changed.

The post-tracing algorithm of RxDeep makes quick reference checks to rule out identical objects, so when making a change respecting maximal object immutability, and with the aforementioned criteria holding (constant kkk leafs are actual subjects of the change), post-tracing consumes O(klog⁡(nδ))\Omicron(k\log(n_{\delta}))O(klog(nδ​)) operations, so the complexity of overall change propagation is given by:

O(log(n)+(k−1)log(nδ))=O(log(n))\Omicron(log(n) + (k - 1)log(n_{\delta})) = \Omicron(log(n))O(log(n)+(k−1)log(nδ​))=O(log(n))1

Additionally, with this solution affected states emit exactly once.
